Visão geral do produto

Declaração do problema

É uma queixa comum dos estudantes universitários a dificuldade na organização de suas atribuições acadêmicas por diversos motivos, como por exemplo: uma jornada dupla feita pelo estudante (estágio/trabalho + jornada acadêmica) e plataformas educacionais confusas e com funcionalidades limitadas. Outro empecilho, para a devida organização, bastante ouvido em grupos estudantis é a dificuldade de armazenar em um só lugar todos os dados e ferramentas necessárias para a organização do aluno e consequentemente sua evolução acadêmica. Uma solução para esse problema poderia ser uma plataforma de fácil utilização para que o estudante possa anotar e administrar qualquer informação pertinente, como por exemplo: contato dos professores, datas de avaliações, notas de avaliações e também contatos de monitores das matérias, etc.

O problema Dificuldade dos alunos se organizarem
Afeta Estudantes
Cujo impacto é Evasão escolar, desânimo quanto a graduação, transtornos de ansiedade
Uma solução de sucesso seria Uma plataforma com uma interface sem poluição visual, com avisos sucintos e claros sobre avaliações, deveres, trabalhos e contatos da rede docente

Declaração de Posição do Produto

Com base no problema exposto acima, foi pensado o MyUni. O MyUni será projetado para que os usuários possam administrar suas tarefas estudantis além de compartilharem informações na área colaborativa (fóruns) da plataforma.

Para Estudantes
Quem Alunos com queixas quanto a organização de vida acadêmica
O MyUni O MyUni será uma plataforma WEB para auxiliar alunos na organização quanto atividades, provas e também um espaço de informações para que eles possam acessar
Que Apresenta funcionalidades que interessam a esses alunos, como: Agenda de contato dos professores, área de criação de notas pessoais, dashboards com alguns índices sobre as turmas, matérias, professores e entre outros (os índices que interessam aos alunos serão coletados por pesquisa realizada posteriormente)
Ao contrário Ao contrário de Plataformas Educacionais focadas em Cadernos/Notas Digitais ou Agenda Digital
Nosso produto Além da Agenda Digital e Cadernos Digitais, nosso produto terá os dashboards com os índices coletados e também uma área colaborativa onde os alunos poderão compartilhar informações sobre determinado professor, turma e matéria

Objetivos do Produto

Com a criação dessa plataforma, o objetivo desejado é trazer facilidades, diminuindo as queixas de estudantes quanto ao quesito organização e de certa forma diminuir o desgaste dos docentes ao responderem perguntas que poderiam ser sanadas se os estudantes tivessem posse da plataforma com a informação unificada em somente um lugar.

Abordagem de desenvolvimento de software

Metodologia

Para o desenvolvimento do software, optamos por seguir com uma metodologia ágil, com essa abordagem, procuramos focar no funcionamento do software em si, por conseguinte, minimizar o risco de entregar um produto que não atenda as nossas expectativas, dividindo-o em curtos períodos para termos uma maior cadência de feedbacks sobre as funcionalidades que estão sendo desenvolvidas, nos possibilitando fazer adaptações em nosso produto, logo tendo maior respostas a mudanças.

Portanto, pensando em um modelo ágil, rápido, adaptável e eficaz, vamos utilizar o Framework Scrum para satisfazer as nossas necessidades. O Scrum, por sua vez, possui a divisão de 3 papéis, o primeiro papel se dirige ao Scrum Master, que é a denominação para o integrante da equipe que ajuda a todos do Time Scrum a entenderem a teoria, prática, regras e valores do Scrum , liderando a equipe utilizando as regras e processos do Scrum, o segundo papel é o Product Owner, ele se concentra na parte comercial e traduz a visão do projeto a equipe, sendo assim responsável pelo Backlog do Produto e por último, mas não menos importante, a equipe, que são os encarregados de desenvolver cada uma das funcionalidades do projeto. A aplicação da metodologia Scrum requer a realização de alguns eventos, como a criação das Sprints, que são as funcionalidades que a equipe deve trabalhar e desenvolver, a definição do backlog do produto, que é o conjunto de sprints ordenadas por prioridade, o planejamento de Sprints, que tem como objetivo discutir como será feito uma sprint a partir da definição do backlog, as daily Scrum, que são reuniões para avaliar as sprints e determinar as alterações dentro de um curto prazo, a revisão de sprints, que é uma revisão do que foi desenvolvido para as entregas futuras e a retrospectiva da sprint, que serve para avaliar o feedback, ver os erros cometidos e identificar possíveis melhorias para o projeto e implementá-las. Além disso, é importante a adoção de um método para gerenciar o fluxo e o tempo de trabalho de cada tarefa que estará sendo executada, assim o método Kanban será aplicado para organizar e equilibrar os processos fazendo com que as atividades não sejam interrompidas por outras.

Assim, a equipe procura se manter fiel a filosofia por trás dos métodos ágeis, que é refletida no manifesto ágil, que foi acordado por muitos dos principais desenvolvedores desses métodos. Esse manifesto afirma:

  • Estamos descobrindo melhores maneiras de desenvolver softwares, fazendo-o e ajudando outros a fazê-lo.
  • Através desse trabalho, valorizamos mais: Indivíduos e interações do que processos e ferramentas Software em funcionamento do que documentação abrangente
  • Colaboração do cliente do que negociação de contrato
  • Respostas a mudanças do que seguir um plano
  • Ou seja, embora itens à direita sejam importantes, valorizamos mais os que estão à esquerda.

Processo

Atividade Objetivo Papel Método Ferramenta
Levantamento de requisitos Nesta etapa será definido todos os requisitos necessários para o funcionamento do produto. Product Owner Será levantado requisitos por meio de mapeamento de problemas que o produto se dispõe para resolver, gerando assim as funcionalidades do produto. Será criado um documento de requisitos com as regras de negócio do produto juntamente com tais funcionalidades de acordo com o mapeamento. Google docs, Meet, Discord.
Organização da metodologia de desenvolvimento Nessa etapa será organizado todas as tarefas com o objetivo de particionar tarefas grandes em tarefas menores e criar um panorama geral dos possíveis impedimentos. Product Owner e Desenvolvedores O Product Owner organizará um backlog com as principais tarefas mapeadas e o time de desenvolvimento irá atribuir para os respectivos responsáveis por essas tarefas em suas respectivas sprints. Ao final de cada sprint será avaliado o desempenho das realizações da sprint e será moldado uma nova sprint de acordo com tal desempenho, refazendo tarefas se preciso. Notion
Prototipação Nesta etapa será projetado o protótipo que funcionará como um guia para o desenvolvimento, de como a aplicação deverá ficar e se comportar. Designer Será criado um protótipo com as telas e fluxos, idealizando a jornada esperada do usuário durante o seu uso na aplicação. Figma, Miro.
Criação dos ambientes de desenvolvimento Nessa etapa deve ocorrer a criação dos ambientes de produção, homologação e desenvolvimento com seus respectivos CDs. DevOps Será criado instâncias EC2 na AWS para cada ambiente de desenvolvimento, disponibilizando chaves SSH para os desenvolvedores. Além disso, será automatizado os deploys com o Github Actions, além de separar os ambientes por branches hospedadas no Github. As instâncias rodaram containers com Docker e os serviços serão organizados com docker-compose. AWS - EC2, Docker, Docker-compose, Github e Github Actions.
Desenvolvimento do back-end Nessa etapa deve ocorrer a criação dos bancos de dados e das APIs que a interface do usuário irá consumir. Desenvolvedores Será criado APIs utilizando Python e FastAPI que consumirão um banco de dados MongoDB. Nessa etapa também será documentado os endpoints das APIs com Swagger, visando facilitar o consumo das APIs em desenvolvimento e ao mesmo tempo criar uma documentação de campos aceitos e quais endpoints cada API oferece. Nessa etapa o pair programming será um método utilizado pela equipe de desenvolvimento, a fim de maximizar aprendizagem, organização e rendimento do desenvolvimento. Python, FastAPI, MongoDB e Swagger.
Desenvolvimento do front-end Com o back-end criado, será feita a construção das interfaces idealizadas na fase de prototipação. Desenvolvedores Começará o desenvolvimento das telas e interfaces utilizando o framework React, que deverá replicar os fluxos idealizados na prototipação, e estabelecer a correta integração com API desenvolvida pelo back-end. Além disso, o pair programming estará presente nessa etapa de desenvolvimento também. Javascript, React Js.
Testagem do produto Nessa etapa será desenvolvido testes unitários, de integração e testes automatizados, testando assim, classes, endpoints e a interface do usuário. Analista de testes Utilizando do método de cobertura de código, será criado testes para todos os escopos do código (Classes, métodos e funções) e testes de integração com Pytest. Além disso, será criado testes automatizados com Selenium. Pytest e Selenium.
Criação de ambiente de análise de dados Nessa etapa será criado o ambiente para análise de dados com objetivo de criar dashboards com dados dos usuários. Desenvolvedores Será integrado o Metabase com o banco de dados da aplicação e será criado dashboards de análise de dados. Metabase

Gerenciamento de Riscos

  • Como no cadastro de usuários existirá distinção de tipos de usuários, existe o risco de criação de diversos usuários por um mesmo agente, com a finalidade de obter acessos indevidos em diversas áreas da aplicação, como áreas direcionadas aos professores.
  • Tentativas de login repetitivas podem dar acesso para agentes indevidos, por meio de scripts de tentativa de acertos de senha.
